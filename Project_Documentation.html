AD Password Reset – Credential Provider Section 1 · Project Overview
Section 1

Project Overview

The AD Password Reset Credential Provider is a self-service password reset solution for Windows domain environments. It adds a custom tile directly to the Windows login / lock screen so that users can reset their own Active Directory password without calling IT support — even before they have logged in.

Problem It Solves

In a standard Active Directory environment, a user who forgets their password must call the IT helpdesk, wait for an admin to reset it manually, and then log in. This process is slow, expensive, and disruptive. This project automates the entire flow securely via email OTP.

What the User Sees

On the Windows lock screen or login screen, the user sees an extra tile labelled "Reset AD Password". Clicking it opens a form with fields for their AD username and a one-time password. After verifying identity via OTP sent to their registered email, they can set a new password immediately.

High-Level Flow

1
User clicks "Reset AD Password" tile

The tile appears on every domain-joined machine (viz.in domain) at the login or lock screen.

2
Credential Provider checks network connectivity

It calls the backend health endpoint. If unreachable, it shows "Connect VPN" to launch SonicWall NetExtender.

3
User enters AD username and clicks "Send OTP"

The backend validates the user exists in Active Directory and sends a 6-digit OTP to their registered email.

4
User enters OTP + new password and clicks "Reset Password"

The backend verifies the OTP, then resets the AD password via LDAP. The OTP is consumed and cannot be reused.

5
Success — log in immediately with new password

Status shows "Password reset successful." The user can now log in with the Windows password tile.

Two Components

💻
Credential Provider DLL (C++)
  • Runs inside Windows LogonUI process
  • Renders the login tile and all UI fields
  • Handles VPN detection and connection
  • Communicates with backend over HTTPS
  • File: ResetPasswordCP.dll
⚙
Backend API (ASP.NET Core 8)
  • Runs as a Windows Service on the server
  • Exposes HTTPS REST API on port 8443
  • Manages OTP generation, email, AD password reset
  • Connects to Active Directory via LDAP over SSL
  • File: ResetPasswordAPI.exe

AD Password Reset – Credential Provider Section 3 · Backend API
Section 3

Backend API (ASP.NET Core 8)

The backend is a single-file ASP.NET Core 8 Minimal API written in C#. It runs as a Windows Service on the domain controller at 172.16.10.4:8443. All traffic is HTTPS-only — no plain HTTP fallback.

File: backend-dotnet/Program.cs

The entire backend is one file, structured in this order:

  1. .env file loader — reads configuration from a .env file next to the exe
  2. OTP store — in-memory dictionary of pending OTP codes
  3. LDAP helpers — user lookup and password reset functions
  4. Email helper — sends OTP email via SMTP
  5. App builder — configures Kestrel, rate limiter, logging
  6. Route handlers — the three HTTP endpoints

API Endpoints

GET /health Network reachability check

Called by the credential provider DLL every 5 seconds to check if the server is reachable. Returns 200 OK. No authentication required.

Response (200):

{ "status": "ok", "time": "2026-05-11T06:30:00.000Z" }
POST /api/auth/request-otp Request a one-time password

Request body:

{ "username": "john.doe" }

What it does:

  1. Validates that username is present
  2. Searches Active Directory for a user with that sAMAccountName who is not disabled
  3. If not found, returns 404 "User not found or account disabled"
  4. Generates a cryptographically random 6-digit OTP, stores it with an expiry timestamp
  5. Sends the OTP to the user's AD-registered email address (the mail attribute)
Status CodeMeaning
200OTP sent to email successfully
400Username missing from request
404User not found in AD or account is disabled
429Rate limit hit (max 10 attempts per 15 minutes)
500AD connection error or SMTP error
POST /api/auth/reset-password Verify OTP and set new password

Request body:

{ "username": "john.doe", "otp": "482931", "newPassword": "MyNewPass@2026!" }

What it does:

  1. Validates all three fields are present; validates new password is at least 12 characters
  2. Verifies OTP matches the stored code and has not expired (one-time use — consumed on first use)
  3. Looks up the user's Distinguished Name (DN) in Active Directory
  4. Resets the password by writing to the unicodePwd LDAP attribute
Status CodeMeaning
200Password reset successfully
400Missing fields, password too short, or AD password policy rejected
401OTP invalid or expired
404User not found in AD
429Rate limit exceeded
500AD connection error

OTP Engine

OTPs are stored in a ConcurrentDictionary<string, (string Code, DateTime Expiry)> keyed by lowercase username.

  • 6-digit numeric code — generated via RandomNumberGenerator.GetInt32(100000, 1000000) (cryptographically secure)
  • Configurable TTL — default 5 minutes (OTP_TTL_SECONDS=300)
  • One-time use — consumed and deleted from the store immediately on validation
  • Automatic expiry — checked on each validation attempt; expired entries are removed
  • Case-insensitive — username is lowercased before use as the dictionary key

LDAP / Active Directory Integration

Two LDAP operations are performed, both over LDAP-SSL (port 636):

User Lookup — LDAP search filter used:

(&(objectClass=user)(sAMAccountName=<username>)(!(userAccountControl:1.2.840.113556.1.4.803:=2)))

The userAccountControl bit-mask check ensures disabled accounts are excluded. Retrieves the mail and displayName attributes.

Password Reset — modifies the unicodePwd attribute with the new password encoded as UTF-16LE wrapped in quotes. This is the standard and only supported method for programmatic AD password resets. It requires LDAPS (SSL).

Rate Limiting

Uses ASP.NET Core 8's built-in rate limiter with a fixed-window policy: maximum 10 requests per IP per 15 minutes on both auth endpoints. Exceeding the limit returns HTTP 429. This prevents brute-force OTP guessing attacks.

Section 4

Credential Provider DLL (C++)

What Is a Windows Credential Provider?

A Credential Provider is a COM in-process DLL that Windows loads into the LogonUI.exe process to display custom authentication tiles on the login screen, lock screen, and UAC prompts. Windows ships built-in credential providers for password, PIN, and fingerprint. This project adds a custom one specifically for AD self-service password reset.

The provider implements two COM interfaces:

  • ICredentialProvider — manages the tile at the screen level (field layout, tile count)
  • ICredentialProviderCredential — manages a single tile's UI fields and user interactions

Source Files

FilePurpose
dllmain.cppDLL entry point, COM exports (DllGetClassObject, DllRegisterServer, DllUnregisterServer), registry write/delete logic
CredentialProvider.h/.cppImplements ICredentialProvider and IClassFactory — manages the tile, validates domain, creates credential object
Credential.h/.cppImplements ICredentialProviderCredential — all UI fields, button handlers, OTP flow, VPN state machine, background monitor thread
HttpClient.hHeader-only HTTP client using WinHTTP: IsApiReachable(), HttpPostJson(), JsonEscape(), JsonExtractString()
ResetPasswordCP.defModule-definition file — explicitly exports the four required COM entry points for the linker

CLSID and COM Registration

Every COM object has a globally unique identifier. This provider's CLSID is:

{B5A84CD4-1F6A-4A52-9A11-D90E6C9D9C4A}

Running regsvr32 ResetPasswordCP.dll calls DllRegisterServer(), which writes three registry entries:

Registry KeyValueEffect
HKCR\CLSID\{...}ResetPasswordCredentialProviderNames the COM object
HKCR\CLSID\{...}\InprocServer32Full path to DLL + ThreadingModel=ApartmentTells Windows where to load the DLL from
HKLM\...\Authentication\Credential Providers\{...}ResetPasswordCredentialProviderThis is what makes the tile appear at the login screen

Domain Guard

In SetUsageScenario(), the provider reads the machine's DNS domain name using GetComputerNameExW(ComputerNameDnsDomain). If the machine is not joined to viz.in, it returns E_NOTIMPL which tells LogonUI to not show the tile. If the lookup fails (e.g., offline transient error), the tile remains visible so the VPN connect button is still accessible.

Usage Scenario Filtering

The tile only appears in two scenarios:

  • CPUS_LOGON — initial login screen
  • CPUS_UNLOCK_WORKSTATION — lock screen unlock

It is hidden for CPUS_CREDUI (UAC prompts) and CPUS_CHANGE_PASSWORD, preventing it from appearing in inappropriate contexts.

UI Field Layout

The tile has 8 fields defined in the global array g_FieldDescriptors[]:

IDLabelTypeVisible When
0Reset AD PasswordLARGE_TEXTAlways (both collapsed and expanded)
1AD UsernameEDIT_TEXTAlways when tile is selected; auto-focused
2Send OTPCOMMAND_LINKOnly when network is connected
3OTP CodePASSWORD_TEXTOnly when network is connected

5StatusSMALL_TEXTAlways; shows live status and error messages
6Connect VPNCOMMAND_LINKOnly when there is NO network connection

VPN State Machine

The credential object tracks network state using a thread-safe volatile LONG _state variable modified via InterlockedExchange():

STATE_INIT -> Initial state. On tile select, immediately probe /health endpoint.
STATE_NO_NETWORK -> API unreachable. Hides OTP fields, shows "Connect VPN" button.
STATE_CONNECTING_VPN -> User clicked "Connect VPN". NEGui.exe launched. Background monitor thread started.
STATE_CONNECTED -> API reachable. OTP fields shown, "Connect VPN" hidden.
STATE_OTP_FLOW -> OTP sent successfully. User is filling in OTP and new password.

Background Monitor Thread

When the user clicks "Connect VPN", a background thread is spawned via CreateThread(). It polls GET /health every 5 seconds for up to 20 attempts (100 seconds total).

Because LogonUI runs in a single-threaded COM apartment (STA), the worker thread must use COM interface marshaling to update the UI safely from a different thread: 

// On the main thread — marshal events interface into a stream
CoMarshalInterThreadInterfaceInStream(
    IID_ICredentialProviderCredentialEvents, _pEvents, &pCtx->pStream);

// On the worker thread — unmarshal it back
CoGetInterfaceAndReleaseStream(pCtx->pStream,
    IID_ICredentialProviderCredentialEvents, &pEvents);

When the VPN connects and the API becomes reachable, the thread updates field visibility directly through the marshaled interface, then exits cleanly. If VPN times out after 100 seconds, it resets state back to NO_NETWORK.

HTTP Client (HttpClient.h)

All HTTP communication uses the Windows built-in WinHTTP library. Key behaviours:

  • TLS only — WINHTTP_FLAG_SECURE is always set; no plain-HTTP fallback
  • Self-signed certificate bypass — relaxes certificate validation for internal certs using SECURITY_FLAG_IGNORE_UNKNOWN_CA | SECURITY_FLAG_IGNORE_CERT_CN_INVALID | SECURITY_FLAG_IGNORE_CERT_DATE_INVALID
  • Short timeouts — /health uses 2-second timeouts; API calls use 5-second timeouts to prevent login screen freezing
  • JSON escaping — JsonEscape() properly escapes quotes, backslashes, newlines in user input before sending
  • JSON parsing — lightweight JsonExtractString() reads error messages from server JSON responses without any JSON library dependency

Secure Memory Handling

Sensitive strings (OTP code, new password) are cleared from memory using SecureZeroMemory() immediately after use. This prevents sensitive data from lingering in process memory pages where it could be read by other processes or appear in memory dumps.

Step 2 — Understanding the COM Interface Chain

A Credential Provider requires implementing a chain of COM interfaces in a specific order:

// 1. Windows calls DllGetClassObject() to get a factory object
CCredentialProviderFactory : public IClassFactory
    CreateInstance() creates a CCredentialProvider

// 2. Windows calls SetUsageScenario() to decide if the tile should appear
CCredentialProvider : public ICredentialProvider
    GetFieldDescriptorCount/At() -- describes the UI field layout
    GetCredentialCount/At()      -- tells Windows there is 1 tile

// 3. Windows calls methods on the credential object for UI interactions
CCredential : public ICredentialProviderCredential
    GetFieldState()      -- which fields are currently visible
    GetStringValue()     -- get text content of a field
    SetStringValue()     -- user typed in a field (username, OTP, password)
    CommandLinkClicked() -- user clicked Send OTP or Connect VPN
    GetSerialization()   -- user clicked the Submit (Reset Password) button

Step 4 — Generate a Self-Signed TLS Certificate

Run on the server (172.16.10.4) in PowerShell as Administrator:

# Create a self-signed certificate valid for 10 years
$cert = New-SelfSignedCertificate `
    -DnsName "172.16.10.4" `
    -CertStoreLocation "cert:\LocalMachine\My" `
    -NotAfter (Get-Date).AddYears(10)

# Export as PFX
$pwd = ConvertTo-SecureString -String "YourPfxPassword" -Force -AsPlainText
Export-PfxCertificate `
    -Cert "cert:\LocalMachine\My\$($cert.Thumbprint)" `
    -FilePath "C:\ResetPasswordAPI\server.pfx" `
    -Password $pwd
Why Self-Signed?
Since the backend is only accessed by the credential provider DLL (not a browser), and the DLL already bypasses certificate validation for internal certs, a self-signed certificate is sufficient for this internal deployment. For production hardening, use a certificate from Active Directory Certificate Services instead.

Deployment Guide

Part A — Deploy the Backend API as a Windows Service

Run these commands on the server (172.16.10.4) in PowerShell as Administrator:

1. Create the service folder

New-Item -Path "C:\ResetPasswordAPI" -ItemType Directory -Force

2. Copy the published files to the server

# From dev machine — copy the publish folder
Copy-Item "d:\reset password\publish\*" `
    -Destination "\\172.16.10.4\C$\ResetPasswordAPI\" -Recurse -Force

# Copy the .env configuration file
Copy-Item "C:\path\to\.env" -Destination "\\172.16.10.4\C$\ResetPasswordAPI\.env"

3. Open firewall port 8443

New-NetFirewallRule `
    -DisplayName "Reset Password API" `
    -Direction Inbound -Protocol TCP `
    -LocalPort 8443 -Action Allow

4. Register and start as Windows Service

New-Service `
    -Name "ResetPasswordAPI" `
    -BinaryPathName "C:\ResetPasswordAPI\ResetPasswordAPI.exe" `
    -DisplayName "AD Password Reset API" `
    -StartupType Automatic

Start-Service -Name "ResetPasswordAPI"
Get-Service  -Name "ResetPasswordAPI"

5. Verify it is working

# Should return: { "status": "ok", "time": "..." }
Invoke-RestMethod -Uri "https://172.16.10.4:8443/health" -SkipCertificateCheck
Service Running Successfully
If you see { "status": "ok" }, the backend is correctly deployed and listening on port 8443.

Part B — Deploy the Credential Provider via Group Policy

The DLL must be copied to C:\Windows\System32\ and registered on every domain machine. A GPO startup script handles this automatically on every domain PC.

Step 1 — Create network share on server

New-Item -Path "C:\IT-Deploy" -ItemType Directory -Force

New-SmbShare -Name "IT-Deploy" -Path "C:\IT-Deploy" `
    -ReadAccess "Domain Computers" -FullAccess "Domain Admins"

$acl  = Get-Acl "C:\IT-Deploy"
$rule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "Domain Computers","Read","ContainerInherit,ObjectInherit","None","Allow")
$acl.AddAccessRule($rule)
Set-Acl "C:\IT-Deploy" $acl

Step 2 — Copy DLL to the share (from dev machine)

net use \\172.16.10.4\IT-Deploy /user:VIZPAY\Administrator *

Copy-Item "d:\reset password\x64\Release\ResetPasswordCP.dll" `
    -Destination "\\172.16.10.4\IT-Deploy\" -Force

Step 3 — Create GPO startup script and link to domain

# Create the deploy batch script (idempotent — skips if already installed)
@'
@echo off
set DLL_SOURCE=\\172.16.10.4\IT-Deploy\ResetPasswordCP.dll
set DLL_DEST=%SystemRoot%\System32\ResetPasswordCP.dll
set REG_KEY=HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Authentication\Credential Providers\{B5A84CD4-1F6A-4A52-9A11-D90E6C9D9C4A}

if %errorlevel% == 0 exit /b 0

$gpo  = New-GPO -Name "Deploy-ResetPasswordCP"

New-Item -Path $dir -ItemType Directory -Force
Copy-Item "C:\IT-Deploy\deploy_resetcp.bat" -Destination $dir -Force

# Register script in GPO scripts.ini
@"
[Startup]
0CmdLine=deploy_resetcp.bat
0Parameters=
"@ | Out-File "\\$dns\SYSVOL\$dns\Policies\$guid\Machine\Scripts\scripts.ini" -Encoding Unicode

AD Password Reset – Credential Provider Section 8 · End-User Flow
Section 8

End-User Flow

Scenario A — User is on the Corporate Network (or VPN already connected)

1
Open the lock screen or login screen

User sees standard login options plus the "Reset AD Password" tile.

2
Click the tile

Status briefly shows "Checking network…". Since the backend is reachable, the OTP form fields appear. Username field is auto-focused.

3
Enter AD Username then click "Send OTP"

Status shows "Sending OTP…". Backend verifies the user exists in AD and sends a 6-digit code to their AD-registered email.

4
Check email on phone or another device

Receives an email from IT Support containing the 6-digit OTP and a notice that it expires in 5 minutes.

5
Enter OTP + New Password then click "Reset Password"

Status shows "Resetting password…". Backend verifies OTP and resets the unicodePwd attribute in Active Directory.

6
Log in immediately with new password

Status shows "Password reset successful. You may now log in." User switches to the password tile and logs in with new credentials.

Scenario B — User is Off-Network (No VPN)

1
Click the "Reset AD Password" tile

Health check fails. Status: "Not connected. Click 'Connect VPN' to launch the VPN client." OTP fields are hidden; only "Connect VPN" button shows.

2
Click "Connect VPN"

SonicWall NetExtender (NEGui.exe) launches from Program Files. Status: "NetExtender opened. Connect VPN then return here." A background thread begins polling /health every 5 seconds.

3
Connect VPN in the NetExtender window

Status updates in real time: "Waiting for VPN… (attempt X of 20)". No action needed on the tile.

4
VPN connects — tile updates automatically

Background thread detects API is reachable. OTP fields appear automatically. Status: "VPN connected. Enter your AD username and click Send OTP."

5
Continue from Scenario A, Step 3

Identical to the on-network flow from this point forward.

AD Password Reset – Credential Provider Section 9 · Security Design
Section 9

Security Design

Security ControlHow It Is Implemented
HTTPS Only The backend uses Kestrel with TLS via PFX certificate. The DLL always sets WINHTTP_FLAG_SECURE. There is no plain HTTP fallback.
OTP Identity Verification Password reset requires proof of email access. Without the OTP delivered to the AD-registered email, an attacker cannot reset a password even if they know the username.
One-Time OTP Each OTP is deleted from the store immediately on first use. Replay attacks are not possible.
OTP Expiry OTPs expire after 5 minutes (configurable). The short window limits exposure if an email is intercepted.
Rate Limiting Maximum 10 OTP requests per source IP per 15 minutes. This prevents brute-force enumeration of usernames or OTP codes.
LDAP Injection Prevention LdapEscape() in Program.cs escapes all special LDAP characters (backslash, asterisk, parentheses, null bytes) before constructing LDAP filter strings.
JSON Injection Prevention JsonEscape() in the DLL escapes all special JSON characters before embedding user input into the HTTP request body.
Disabled Account Exclusion The LDAP filter includes a userAccountControl bit-mask check to ensure disabled accounts cannot receive OTPs or have their password reset.
Secure Memory Clearing SecureZeroMemory() is called on password and OTP string buffers in the DLL immediately after use, preventing sensitive data from lingering in memory dumps.
Domain-Restricted Tile The tile only appears on machines joined to the viz.in domain. Machines on other domains or workgroups do not see the tile.
Least Privilege Service Account The LDAP bind account should have only Read and Reset Password delegated permissions — not Domain Admin or full AD write access.
LDAP over SSL (LDAPS) All LDAP communication uses port 636 with SSL. AD requires LDAPS for unicodePwd modifications — plain LDAP rejects them outright.
Short HTTP Timeouts Health check uses 2-second timeouts; API calls use 5-second timeouts. This prevents the Windows login screen from freezing if the server is unreachable.
Known Limitation: Self-Signed Certificate
The DLL bypasses TLS certificate validation to support internal self-signed certificates. For production hardening, issue a certificate from Active Directory Certificate Services and remove the SECURITY_FLAG_IGNORE_UNKNOWN_CA bypass flags from HttpClient.h.
Known Limitation: In-Memory OTP Store
OTPs are stored in process memory only. If the service restarts, all pending OTPs are lost and users must request a new code. For a high-availability environment, consider persisting OTPs to a database with TTL expiry.

Section 10

Troubleshooting

Credential Provider DLL Issues

ProblemCauseFix
Tile does not appear on login screen DLL not registered, wrong domain, or wrong platform (32-bit vs 64-bit) Run regsvr32 C:\Windows\System32\ResetPasswordCP.dll as Administrator. Verify machine is joined to viz.in domain. Confirm the Credential Providers registry key exists. Reboot.
Tile shows but immediately says "Not connected" Backend not running or firewall blocks port 8443 Start the ResetPasswordAPI service. Run Test-NetConnection 172.16.10.4 -Port 8443. Check firewall rule exists.
regsvr32 fails with "access denied" Not running as Administrator Right-click PowerShell and choose "Run as Administrator" before running regsvr32.
GPO script does not run on new machines GPO not yet applied Run gpupdate /force and reboot. Check GPO is linked and enabled in Group Policy Management Console.

Backend API Issues

ProblemCauseFix
Service fails to start Missing or invalid PFX file Check TLS_PFX_PATH in .env points to an existing PFX. Check the password. Check Windows Event Log: Application source ResetPasswordAPI.
"Failed to contact directory server" LDAP connection to AD fails Run Test-NetConnection 172.16.10.4 -Port 636. Verify AD_BIND_DN and AD_BIND_PASSWORD are correct.
"Could not send OTP email" SMTP settings wrong or server unreachable Verify SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD. Test SMTP connectivity from the server.
"Password does not meet domain complexity requirements" AD password policy rejected the new password Use a longer password meeting your domain's complexity rules. The API requires a minimum of 12 characters.
OTP emails not arriving User has no mail attribute in AD, or email in spam Check the user's mail attribute is populated in Active Directory Users and Computers. Check the spam/junk folder.
429 Too Many Requests Rate limit exceeded Wait 15 minutes. For testing, restart the service to reset the rate limiter state.
